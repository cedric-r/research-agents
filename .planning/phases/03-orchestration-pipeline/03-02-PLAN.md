<objective>
**As a** user, **I want to** never have the research session blocked by a single hung agent, **so that** I always get results back within a bounded time, with partial answers from agents that couldn't complete in time.

This plan adds the complete 4-layer timeout architecture: Layer 1 (documented inactive for CLI, D-13), Layer 2 (already implemented in HttpHelper from Phase 2 -- confirmed CURLOPT_TIMEOUT active, D-14), Layer 3 (deferred to v2, D-15), and Layer 4 (cooperative agent-step deadline, D-16). It implements per-step timeout enforcement in the Arbitrator using pcntl_alarm + SIGALRM with per-child deadline tracking, a flag-based SIGTERM handler in forked children (D-10), and a cooperative deadline check in ResearchAgent.

**Purpose:** Prevent any single hung agent or API call from blocking the entire research session. Guarantee bounded response time.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| parent process -> child process (SIGTERM) | Parent sends SIGTERM to child when timeout expires |
| child process -> signal handler | SIGTERM handler in child sets flag -- critical: NO file I/O in handler |
| parent process -> child process (SIGKILL) | Child that ignores SIGTERM is force-killed |
| parent process -> SIGALRM | pcntl_alarm fires SIGALRM in parent, handler sets batch timeout flag |
| child process -> filesystem (partial write) | Child writes partial answer to temp file from main context (safe) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | DoS | SIGALRM handler deadlock | mitigate | Handler sets boolean flag only -- no I/O, no complex logic, no mutex/blocking calls. Deferred handling in main loop. |
| T-03-08 | DoS | SIGTERM in child while writing temp file | mitigate | Child's SIGTERM handler sets flag only (async-signal-safe). WriteTempFile in main context uses LOCK_EX. If SIGTERM arrives mid-write, partial write is detected by JSON decode failure in parent. |
| T-03-09 | Tampering | False partial answer (child writes before signal) | accept | The child's partial answer is its honest best effort before timeout. No integrity attack vector -- child process is the agent it claims to be. |
| T-03-10 | DoS | Zombie from killed child not reaped | mitigate | Both SIGTERM and SIGKILL paths call pcntl_waitpid($pid, $status) to reap the child. The SIGCHLD safety-net handler reaps any missed zombies. |
| T-03-11 | Spoofing | Inherited SIGALRM handler in child | mitigate | Child explicitly calls pcntl_signal(SIGALRM, SIG_DFL) and pcntl_alarm(0) after fork (per RESEARCH.md Pitfall 3). This prevents parent's alarm setup from affecting child execution. |
| T-03-12 | DoS | Layer 1 inactive (CLI: max_execution_time = 0) | accept | D-13: explicit user decision to rely on existing PHP defaults. Layers 2, 4, and batch alarm provide sufficient coverage. Layer 2 (HTTP socket timeout = 60s) is the primary guard against long-running operations. |
| T-03-13 | Denial of Service | signal handler writes to temp file | mitigate | SIGTERM handler is strictly flag-based. The handler body contains ONLY `$timedOut = true;` -- no file_put_contents, no complex logic. RESEARCH.md confirms file_put_contents is not async-signal-safe. |
| T-03-SC | Tampering | Package legitimacy | accept | No new Composer packages. Existing phpunit/phpunit in require-dev from Phase 2. |
</threat_model>
